# ArtiChat 0.1.3 Milestone 4 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Execution constraint:** Execute inline; do not use subagents, push GitHub, or touch production.

**Goal:** Add registration controls, SMTP configuration, customizable mail, verification-code login, password reset, and email verification for sensitive account actions.

**Architecture:** Store non-secret switches in Config, encrypt SMTP credentials with a Fernet key derived from `WEBUI_SECRET_KEY`, and keep verification challenges/reset tokens in dedicated additive SQLAlchemy tables. A single mail service renders whitelisted variables into subject/Markdown templates, records deliveries, and sends through stdlib SMTP in a worker thread. Auth routes call the verification service; OAuth and LDAP remain unchanged.

**Tech Stack:** FastAPI, Pydantic, SQLAlchemy async, Alembic, cryptography/Fernet, hashlib/secrets, smtplib/email.message, Markdown, pytest, SvelteKit.

## Security Invariants

- Codes are six digits, valid ten minutes, resend cooldown sixty seconds, max five sends per email/hour, max five attempts, and consumed immediately on success.
- Reset links are valid thirty minutes and single use.
- Public request responses never reveal whether an email exists.
- SMTP passwords, codes, and reset tokens never appear in normal logs or API responses.
- Existing users remain able to password-login after upgrade; OAuth/LDAP are not routed through local email verification.
- Initial admin bootstrapping cannot be locked out by missing SMTP configuration.
- Business email failure never rolls back a committed subscription change.
