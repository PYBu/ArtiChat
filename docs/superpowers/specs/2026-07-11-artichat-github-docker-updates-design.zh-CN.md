# ArtiChat GitHub Docker 更新系统设计

## 目标

为已经通过 Docker 部署到 AWS/1Panel 服务器的 ArtiChat 增加可控更新能力：开发者发布新版本后，管理员可在“管理员设置 -> 通用 -> 版本”中检测更新，并在确认后完成镜像下载、数据备份、容器重建、健康检查和失败回滚。

正常更新允许短暂停机，不要求零停机。更新不得用本地数据库覆盖服务器数据库，也不得影响同一服务器上的其他容器。

## 已确认决策

- 使用公开 GitHub 仓库，但首次推送使用清理后的代码快照，不公开现有内部 Git 历史。
- 使用语义化版本标签发布，例如 `v0.1.2`。
- GitHub Actions 构建 Docker 镜像并发布到公开 GHCR。
- 发布标签只产生可部署版本，不立即更新服务器。
- 管理员在 ArtiChat 页面确认后，才触发独立的 GitHub Actions 部署工作流。
- 服务器不构建镜像，只拉取 GitHub Actions 已构建并测试的版本镜像。
- 保持服务器数据挂载 `/data/artichat-prod/data:/app/backend/data` 不变。
- 保留最近一个可回滚镜像和最近三份部署前数据备份。

## 非目标

- 不实现零停机或多节点滚动发布。
- 不允许 ArtiChat 容器直接访问 `/var/run/docker.sock`。
- 不在服务器上执行前端或 Python 依赖构建。
- 不通过更新流程迁移或覆盖本地开发数据库。
- 不允许管理员输入任意镜像地址、Git 引用或 Shell 命令。
- 不自动合并代码或自动创建版本标签。

## 总体架构

更新系统由四部分组成：

1. **GitHub 发布工作流**：验证标签和项目版本，运行测试，构建版本镜像，推送 GHCR，并创建 GitHub Release。
2. **ArtiChat 更新服务**：查询自己的 GitHub Releases，比较当前版本与最新版本，验证管理员部署请求，并触发部署工作流。
3. **管理员版本界面**：显示当前版本、最新版本、更新说明、发布时间、部署状态以及部署按钮。
4. **服务器部署脚本**：在固定目录内拉取固定仓库的版本镜像，备份运行数据，重建 ArtiChat 容器，执行健康检查并在失败时回滚。

ArtiChat 容器只调用 GitHub API，不直接控制 Docker。Docker 操作由 GitHub Actions 通过受限 SSH 身份在服务器上执行。

## 版本与发布

### 版本来源

Git 标签是发布版本的最终来源。发布工作流必须验证以下版本一致：

- 标签版本，例如 `v0.1.2`。
- `package.json` 与 `package-lock.json` 的项目版本。
- 后端返回的运行版本。
- 前端显示版本和 changelog 版本。

构建时把版本号和 Git 提交哈希写入镜像。运行中的 `/api/version` 返回机器可比较的纯版本号以及构建哈希；`Artivis Alpha` 等展示名称与纯版本号分开。

### 发布工作流

创建 `vX.Y.Z` 标签后，GitHub Actions 顺序执行：

1. 校验标签格式和项目版本一致性。
2. 运行品牌守卫、订阅守卫、更新系统测试和前端构建。
3. 构建 ArtiChat Docker 镜像。
4. 向首次引导时固定配置的 ArtiChat GHCR 仓库推送不可变版本标签 `X.Y.Z`。
5. 可选更新方便人工使用的 `latest` 标签，但部署系统不得依赖 `latest`。
6. 创建 GitHub Release，并使用发布时填写的更新说明。

只有全部检查成功的版本才会出现在管理员更新界面。发布工作流不连接生产服务器。

## 管理员版本界面

复用现有“管理员设置 -> 通用 -> 版本”区域，不改动“帮助”和“授权”区域的职责。

版本区域显示：

- 当前安装版本和构建哈希。
- 最新可用版本、发布时间和发布说明。
- 镜像是否已经构建完成。
- 当前部署状态：空闲、等待、准备、拉取镜像、备份、重启、健康检查、成功或失败。
- 有更新且没有其他更新任务时显示“部署更新”。
- 失败时显示精简错误原因和当前实际运行版本。

点击部署后立即禁用重复操作。服务重启期间，前端轮询 `/health`；服务恢复后重新读取 `/api/version` 和更新状态。成功条件是健康检查通过且运行版本等于目标版本。

## 后端更新服务

后端提供管理员专用接口：

- 查询当前版本与 GitHub 最新 Release。
- 查询当前部署状态。
- 请求部署指定版本。

部署请求必须满足：

- 当前用户是管理员。
- 目标符合严格的 `X.Y.Z` 版本格式。
- 目标对应已发布且镜像构建成功的 GitHub Release。
- 目标高于当前版本。
- 当前没有其他部署任务。

通过后，后端生成不可预测的操作 ID，记录目标版本和等待状态，并用仅限该仓库的 GitHub 凭据触发部署工作流。凭据保存在服务器 `.env`，不会传给浏览器、写入日志或提交到 Git。

版本检查失败时返回当前运行版本和可恢复的错误状态，不影响聊天等主要功能。GitHub Release 查询使用短时间缓存，避免频繁访问 GitHub API。

## 部署工作流

部署工作流接受固定格式的目标版本和操作 ID。它通过专用 SSH 密钥调用服务器上的固定部署脚本，不执行来自页面的任意命令。

服务器部署顺序：

1. 获取部署互斥锁并验证目标版本。
2. 检查数据目录、Compose 文件、当前容器状态和剩余磁盘空间。
3. 从首次引导时固定配置的 ArtiChat GHCR 仓库拉取 `X.Y.Z`，此时旧容器继续提供服务。
4. 记录当前镜像版本，作为本次回滚目标。
5. 停止 ArtiChat 容器。
6. 对 `/data/artichat-prod/data` 创建一致性备份，备份写入 `/data/artichat-prod/backups`。
7. 更新服务器部署配置中的不可变镜像版本。
8. 重新创建 ArtiChat 容器，不修改数据挂载路径和 `WEBUI_SECRET_KEY`。
9. 在限定时间内轮询 `http://127.0.0.1:13000/health` 和 `/api/version`。
10. 成功后记录完成状态，保留上一个镜像并清理更早的部署镜像和超出三份的备份。

部署状态使用原子写入方式保存在 `/data/artichat-prod/data/update-state/status.json`，确保容器重启后仍可读取。业务数据备份排除 `update-state`，回滚完成后由部署脚本写入最终状态。状态文件不包含 GitHub 令牌、SSH 信息或其他秘密。

## 数据保护与回滚

镜像和运行数据严格分离。重建容器只替换程序层；SQLite 数据库、上传文件、配置、聊天记录、订阅、兑换码、公告和知识库继续位于 `/data/artichat-prod/data`。

备份在容器停止后创建，避免复制正在写入的 SQLite 数据库。备份写入 `/data` 分区，不占用空间较紧张的根分区。

如果新容器未通过健康检查或版本不匹配，部署脚本自动：

1. 停止失败的新容器。
2. 恢复部署前数据备份，撤销可能已经执行的数据库迁移。
3. 把 Compose 镜像恢复为上一个版本。
4. 重新创建旧容器并验证健康状态。
5. 记录失败阶段、简化错误和最终运行版本。

自动恢复仅适用于本次部署尚未成功对外恢复服务的情况。已成功运行后发生的业务故障不自动恢复旧数据库，避免覆盖更新后产生的新数据。

## 安全边界

- 公开仓库首次推送前执行秘密扫描，并清理服务器 IP、用户名、备份文件名和内部部署记录。
- 保留并遵守现有 Open WebUI 许可证和多许可证声明。
- GitHub Actions Secrets 保存服务器地址、专用 SSH 私钥和部署所需秘密。
- 服务器 `.env` 保存用于触发单一仓库部署工作流的受限 GitHub 凭据。
- 部署接口只允许管理员访问，并使用互斥锁和幂等操作 ID 防止重复触发。
- 部署脚本固定镜像仓库、Compose 项目、容器名、端口和数据目录。
- 版本输入通过严格正则验证并再次与 GitHub Release 核对。
- SSH 密钥使用受限账户；条件允许时通过 `authorized_keys` 强制只运行 ArtiChat 部署脚本，并禁用端口转发和交互式终端。
- 工作流和部署日志对令牌、密钥、环境文件内容以及敏感命令参数做脱敏。

## 首次引导部署

现有生产版本不包含更新服务，因此需要一次手动引导：

1. 创建公开 GitHub 仓库。
2. 从当前代码生成不包含内部历史和敏感部署记录的初始提交。
3. 配置 GitHub Actions、GHCR、Actions Secrets 和分支保护。
4. 在服务器安装受限部署脚本和专用 SSH 公钥。
5. 在服务器 `.env` 配置仓库信息和受限 GitHub 部署凭据。
6. 把生产 Compose 改为使用不可变 GHCR 镜像标签。
7. 备份当前数据后，手动部署第一个包含更新系统的版本。
8. 验证检测更新、触发部署、状态轮询、重启、数据保留和失败回滚。

首次引导完成后，正常版本更新不再需要登录服务器。SSH、GitHub 或磁盘故障仍保留人工维护路径。

## 测试与验收

### 自动测试

- 版本比较和标签格式测试。
- GitHub Release 响应、缓存和错误处理测试。
- 管理员权限与未授权访问测试。
- 非法版本、未发布版本和降级请求拦截测试。
- 重复点击、并发部署和幂等操作测试。
- GitHub 工作流触发成功、超时和失败测试。
- 部署状态持久化与敏感字段排除测试。
- 发布版本一致性守卫。
- 现有品牌守卫、订阅测试和前端构建。

### Docker 集成验证

- 使用测试数据目录启动当前版本并写入代表性数据。
- 发布或加载测试版本镜像并执行完整部署脚本。
- 验证容器被重建、版本发生变化、数据保持一致且 `/health` 成功。
- 使用故意失败的镜像验证旧镜像和数据备份可以自动恢复。
- 验证服务器上其他容器和 1Panel 服务未被停止或修改。
- 验证备份只保留三份，镜像至少保留当前和上一个版本。

### 生产验收

- 首次引导部署前创建可人工恢复的完整数据备份。
- 从管理员页面完成一次真实的小版本更新。
- 确认服务恢复后显示目标版本。
- 确认账号、聊天、模型配置、订阅、额度、公告、兑换码和上传数据未变化。
- 确认失败路径不会执行带数据卷的 Docker 清理命令。

## 运维约定

- 正常版本只通过 `vX.Y.Z` 标签发布。
- 不重新使用或覆盖已经发布的版本标签。
- 生产 Compose 始终引用不可变版本镜像，不以 `latest` 作为实际部署来源。
- 发布说明由维护者填写，管理员页面直接显示该 Release 内容的安全渲染版本。
- 数据库迁移优先保持向后兼容；即使支持自动恢复，也必须在破坏性迁移前进行专项评审。
- 任何清理命令都不得包含 `--volumes`，也不得影响非 ArtiChat 容器、镜像或数据目录。
