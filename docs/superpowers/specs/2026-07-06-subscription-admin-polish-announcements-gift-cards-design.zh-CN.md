# 订阅后台优化、公告与礼品卡设计

日期：2026-07-06

## 目标

本轮在现有 ArtiChat 订阅系统上补齐运营后台能力和用户侧体验：

- 左下角订阅额度环状图实时刷新，并调整为更小、更细、订阅名称在左侧。
- 订阅名称保持 `Free / Plus / ChatPower`，不再展示“免费版”。
- 订阅卡片由后台可编辑内容驱动，展示介绍、每月额度、可访问模型说明和 icon。
- 新增公告弹窗，管理员可编辑公告并指定弹出规则。
- 默认额度调整为 `Free 100`、`Plus 3000`、`ChatPower 10000`。
- 兑换码支持自定义、完整显示和删除/停用。
- 管理员订阅与用量界面支持按用户 ID、用户名、邮箱、显示名搜索，并优先显示用户邮箱。
- 新增礼品卡发放，管理员可向指定用户或全体用户发放待领取兑换码。

## 现状与根因

`SubscriptionQuotaRing.svelte` 只在 `onMount` 调用一次 `/subscriptions/me`，聊天扣费、兑换成功、后台调整余额后没有共享订阅状态或刷新事件，所以额度不会实时更新。

订阅计划表已有 `description` 和 `features` 字段，可承载用户订阅卡片展示内容，不需要新增营销内容表。

当前 `redemption_code` 只保存 `code_hash` 和 `code_preview`，无法从旧数据还原完整兑换码。后续新建兑换码会新增完整码字段并在后台展示。

公告需要按用户记录已查看状态，不能复用现有顶部 `banners`，因为 banners 只有本地关闭/记忆能力。

礼品卡是“管理员发放给用户的兑换码权益”，需要记录发放对象和领取状态，不能只用普通兑换码表表达。

## 方案

### 订阅状态刷新

新增前端订阅 store，提供：

- `subscription`：当前用户订阅快照。
- `refreshSubscription()`：拉取 `/subscriptions/me`。
- `notifySubscriptionChanged()`：广播订阅已变化。

触发刷新位置：

- 应用加载后初始化一次。
- 聊天 `chat:completion` 完成或 `chat:active=false` 后刷新一次。
- 兑换码兑换成功后刷新。
- 礼品卡领取成功后刷新。
- 用户打开订阅/用量页时刷新。
- 窗口重新聚焦时兜底刷新。

环状图只展示余额比例和颜色，订阅名称放在环左侧。余额耗尽时使用红色。

### 订阅计划展示内容

继续使用 `subscription_plan.description` 和 `subscription_plan.features`。

`features` 约定为对象：

```json
{
  "icon": "sparkles",
  "subtitle": "基础体验",
  "highlights": ["每月 100 Chatpoint", "可访问基础模型"],
  "model_summary": "适合轻量对话和试用模型。",
  "cta_label": "当前计划"
}
```

管理员“订阅计划”页面新增展示内容编辑区，不再要求管理员直接写 JSON。用户订阅页从公开计划接口读取计划卡片，展示后台配置内容和当前额度。

默认计划：

- Free：100 Plan Chatpoint / 30 天
- Plus：3000 Plan Chatpoint / 30 天
- ChatPower：10000 Plan Chatpoint / 30 天

系统只更新仍处于旧默认值的计划，不覆盖管理员手动改过的计划。

### 公告

新增表：

- `announcement`：公告主体。
- `announcement_view`：用户查看记录。

公告字段：

- 标题、正文、icon、按钮文案。
- `display_mode`：`once`、`every_login`、`new_user`。
- 启用状态、开始时间、结束时间、排序、创建者。

弹出规则：

- `once`：用户点击确认后写入查看记录，之后不再弹。
- `every_login`：服务端返回启用公告，前端用 sessionStorage 保证同一次会话只弹一次；下次登录/新会话再弹。
- `new_user`：公告发布后注册的用户可见，点击确认后不再弹。

前端在应用布局加载后拉取待展示公告，使用现有 `Modal` 风格弹窗。后台在订阅设置中新增“公告”页签用于管理。

### 兑换码

`redemption_code` 新增 `code` 字段保存完整兑换码。新建时支持：

- 自动生成。
- 管理员自定义单个兑换码。

后台列表显示完整码；旧兑换码没有完整码时显示原 preview 并标注“旧兑换码未保存完整内容”。

删除采用软删除：把 `is_active=false` 并隐藏为删除状态，保留已兑换记录和账本审计。用户兑换时仍按 hash 校验。

### 礼品卡

礼品卡底层仍是兑换码权益，但新增发放表 `gift_card_grant`：

- `redemption_code_id`
- `user_id`
- `status`：`pending`、`claimed`、`revoked`
- `claimed_at`
- `batch_id`
- `created_by`
- `created_at`

管理员创建礼品卡批次时可选择：

- 指定用户：按邮箱、用户名、显示名或 ID 搜索后选择多个用户。
- 全体用户：发放给当前已有的所有 user/admin 账号。

全体发放只覆盖发放时已有用户，不自动包含未来注册用户。

用户兑换码页面新增“待领取礼品卡”提示卡。点击领取后服务端校验该用户拥有 pending grant，然后调用现有兑换逻辑完成订阅/额度发放，并把 grant 标记为 claimed。

管理员可查看礼品卡批次、发放对象、领取状态、已领取/未领取数量，并可撤销未领取礼品卡。

### 管理员用户显示与搜索

订阅、用量、账本接口返回用户资料：

```json
{
  "user": {
    "id": "...",
    "email": "user@example.com",
    "username": "handle",
    "name": "User Name"
  }
}
```

管理员 UI 主显示邮箱，副显示用户名/显示名。搜索匹配用户 ID、邮箱、用户名、显示名。

## 测试

后端新增/更新订阅测试：

- 默认计划额度和 Free 名称。
- 自定义兑换码创建、完整码展示、兑换、删除后不可兑换。
- 礼品卡指定用户发放、领取、重复领取阻止、撤销后不可领取。
- 全体用户发放只发给已有用户。
- 公告三种展示模式和已查看规则。
- 管理员订阅搜索支持邮箱/用户名/显示名。
- 管理员用量和账本返回用户邮箱资料。

前端增加静态检查脚本覆盖关键文件：

- 不再出现“免费版”作为订阅名称。
- 环状图使用共享订阅刷新。
- 兑换码页包含待领取礼品卡提示。
- 管理员订阅页包含公告、礼品卡、展示内容入口。

## 非目标

- 不实现真实支付。
- 不为礼品卡做邮件/站内信推送。
- 不把全体礼品卡自动发给未来新用户。
- 不还原历史兑换码完整内容。
