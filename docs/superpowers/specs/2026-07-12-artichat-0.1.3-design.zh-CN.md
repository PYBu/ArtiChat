# ArtiChat 0.1.3 设计稿

## 目标

ArtiChat 0.1.3 在现有 0.1.2 基线上完成以下更新：

- 修复模型权限逐行保存、公告删除后仍显示、订阅二级菜单在手机端不可见、暗色主题使用错误 Logo 四项问题。
- 将模型计费升级为类似 OpenAI API 的四段 Token 定价，但使用 `Chatpoint / 1M Token` 作为内部计费单位。
- 重做管理员和用户用量账本，记录四类 Token、消耗、延迟、时间及管理员审计 IP。
- 增加平台设置、注册管理、SMTP 邮箱配置、验证码、邮箱登录和找回密码。
- 将用户订阅和额度管理合并到管理员用户页面。
- 增加用户侧“可领取的礼品”入口。
- 完成发布前文件审计、公开导出检查和 `v0.1.3` 发布准备。

0.1.3 作为一个版本发布，但实现拆分为五个可独立测试和提交的里程碑。

## 已确认的产品决策

- 邮箱系统只使用 SMTP 发信，不支持 IMAP 收信。
- 邮箱验证码登录是密码登录的备选入口，密码登录继续保留。
- OAuth 和 LDAP 登录保持现有流程，不强制通过本地邮箱验证码。
- 敏感操作邮箱验证成功后形成 10 分钟有效窗口。
- 注册域名使用白名单；白名单为空时允许所有合法邮箱。
- 模型价格单位为 `Chatpoint / 1M Token`，不引入美元、人民币或汇率换算。
- 用户端不显示请求 IP；管理员用量账本显示完整审计 IP。
- 0.1.3 不接入 Stripe，也不保存银行卡或第三方支付账户，只保护现有账单地址修改。
- 待领取礼品入口不显示数量，固定文案为“可领取的礼品”。
- 所有功能完成和审查通过后才准备并推送 GitHub 发布；生产更新单独确认。

## 总体架构

0.1.3 继续在 `artichat-brand-whitelabel` 工作树和现有 SvelteKit、FastAPI、SQLAlchemy、Alembic 架构上开发，不创建新的产品代码库。

功能按领域拆分：

1. **计费域**：模型价格、Token 标准化、Chatpoint 计算和余额扣减。
2. **用量域**：不可覆盖的模型请求用量记录、延迟、状态和审计信息。
3. **订阅域**：套餐、有效期、Plan/Check Chatpoint 和余额流水。
4. **邮件域**：SMTP 连接、模板渲染、发送记录、失败原因和重试。
5. **验证域**：验证码、重置链接、限流、失败次数和二次验证窗口。
6. **平台配置域**：平台名称、主题 Logo、About、注册规则和邮箱设置。
7. **后台页面域**：独立运营路由和移动端可用的信息架构。

业务代码不得直接拼接 SMTP 消息或重复实现验证码规则。聊天中间件只向计费域提交标准化用量，不直接维护供应商专用计费公式。

## Bug 修复设计

### 模型权限统一保存

当前模型权限页面逐行调用保存接口。0.1.3 改为：

- 所有模型在前端形成可编辑草稿。
- 页面只保留一个“保存更改”按钮。
- 前端显示是否存在未保存修改。
- 后端提供批量更新接口并使用单个数据库事务。
- 任意模型的权限、计费模式或价格不合法时，整批拒绝保存。
- 成功后返回标准化后的全部模型策略。

### 公告删除

公告继续使用软删除，以保留审计和历史查看能力。修复规则：

- 删除将公告设为停用状态。
- 管理员默认列表不返回已停用公告。
- 页面提供“显示已停用”筛选，供需要时查看历史记录。
- 删除成功后立即从默认列表移除。
- 用户活跃公告接口继续只返回有效公告。

### 手机端订阅管理菜单

以下功能从管理员设置中的订阅二级菜单拆出，成为独立页面和路由：

- 订阅计划
- 模型权限与计费
- 兑换码
- 礼品卡
- 公告
- 用量账本

桌面端使用管理员导航进入；手机端使用单级页面导航和返回路径，不依赖横向滚动或固定宽度二级侧栏。

### 主题 Logo

统一品牌资源选择逻辑：

- 亮色主题使用暗色 Logo。
- 暗色主题使用白色 Logo。
- 左上角侧边栏、登录页、加载页和其他主要品牌入口使用同一主题资源解析器。
- 不在组件中继续写死 `favicon.png` 作为所有主题的品牌 Logo。

## 四段 Token 计费

### 模型策略

每个模型保留：

- `allowed_tiers`
- `quota_mode`：`metered` 或 `unlimited`

并新增四项非负价格：

- `input_chatpoint_per_million`
- `output_chatpoint_per_million`
- `cache_creation_chatpoint_per_million`
- `cache_read_chatpoint_per_million`

价格使用十进制字符串传输和保存，计算时使用 `Decimal`。历史用量必须保存本次实际使用的价格快照，后续修改模型价格不能改变历史账本。

### 标准化 Token

供应商 usage 统一转换为四类互不重复的可计费 Token：

- 普通输入 Token
- 输出 Token
- 创建缓存 Token
- 读取缓存 Token

对于把 cached tokens 包含在总输入中的供应商，标准化层必须从普通输入中扣除对应缓存 Token，避免重复计费。原始 usage 保存到元数据用于排查，但计费只使用标准化字段。

### 计费公式

```text
cost_chatpoint = (
  input_tokens * input_price
  + output_tokens * output_price
  + cache_creation_tokens * cache_creation_price
  + cache_read_tokens * cache_read_price
) / 1_000_000
```

最终继续转换成现有 `cp_micros` 整数单位并向上取整，防止浮点误差和少扣费用。

扣费顺序保持现有规则：

1. 先扣 Plan Chatpoint。
2. 不足部分扣 Check Chatpoint。
3. 当前请求允许导致余额为负。
4. 后续按量请求在总余额不大于零时拒绝。

### 特殊状态

- `unlimited` 模型：记录 Token 和延迟，费用为零。
- 管理员调用：记录 Token 和延迟，费用为零并标记管理员绕过。
- usage 完全缺失：标记 `missing_usage`，费用为零。
- 只有缓存字段缺失：缺失字段按零处理，不影响其他字段计费。

### 旧倍率迁移

现有倍率 `usage_multiplier` 对应旧规则中每百万 Token 的等价费用。迁移时生成等价的输入和输出价格，以保持升级前后的默认成本一致。缓存创建和读取价格默认从零开始，等待管理员按实际供应商价格配置。

旧倍率字段在 0.1.3 中保留兼容，不立即删除；新代码只使用四段价格计费。

## 新版用量账本

### 模型用量记录

每次模型请求生成一条不可覆盖的记录，至少包含：

- 用户 ID 和管理员可读用户摘要
- 模型 ID
- 聊天 ID、消息 ID、请求 ID
- 当前订阅档位和计费模式
- 四类 Token
- 四项价格快照
- 总消耗、Plan 消耗、Check 消耗
- 扣费后的 Plan 和 Check 余额
- 首字延迟毫秒数
- 总耗时毫秒数
- 状态
- 管理员审计 IP
- 原始 usage 元数据
- 创建时间

首字延迟以流式响应首次有效内容为准。无法可靠测量首字时间的非流式或异常请求保存为空，不用总耗时伪造首字时间。

审计 IP 默认使用请求连接地址。只有在部署明确配置可信代理时才解析转发头，防止客户端伪造 `X-Forwarded-For`。

### 管理员页面

管理员用量页面显示：

- 用户
- 模型
- 四类 Token
- 消耗额度
- 首字延迟和总耗时
- 时间
- 完整审计 IP
- 状态

支持按用户、模型、状态和时间筛选。余额变更流水与模型用量分别展示，避免把套餐调整和模型请求混为一类记录。

### 用户页面

用户用量页面显示自己的：

- 模型
- 四类 Token
- 消耗额度
- 首字延迟和总耗时
- 时间和状态

用户页面不显示 IP，也不返回其他用户信息。

历史记录缺少缓存 Token、延迟或 IP 时显示为“不可用”，不得生成虚构数据。

## 注册管理

新增注册管理页面：

- 邮箱域名白名单。
- 是否允许白名单域名的子域名。
- 是否开启注册邮箱验证。
- 是否允许邮箱验证码登录。
- 是否开启敏感操作邮箱验证。

规则：

- 白名单为空时允许所有合法邮箱。
- 域名转小写并进行规范化比较。
- 添加 `example.com` 默认允许 `user@example.com`。
- 开启子域名选项后同时允许 `user@team.example.com`。
- 管理员手动创建用户不受白名单限制。
- 已注册用户不因后续白名单变化被禁用。
- 新注册用户继续由现有订阅生命周期初始化为 Free，不在注册管理中增加默认赠送额度或额外配额配置。

## SMTP 邮箱配置

新增独立邮箱设置页面：

- SMTP 主机
- 端口
- 用户名
- 密码
- TLS/SSL 模式
- 发件人地址
- 发件人名称
- Reply-To
- 平台访问地址
- 邮件功能总开关
- 订阅通知开关

SMTP 密码等敏感配置通过 `WEBUI_SECRET_KEY` 派生密钥加密存储。管理接口读取时只返回是否已配置和掩码，不返回原始密码。

测试分为两步：

1. **连接测试**：DNS、TCP、TLS 和 SMTP 认证，不发送邮件。
2. **测试邮件**：管理员输入收件地址，发送真实的 ArtiChat 测试邮件。

测试结果返回明确阶段和安全错误摘要，不在日志或响应中泄露密码。

## 验证码与账户安全

### 通用验证码规则

- 6 位数字。
- 10 分钟有效。
- 60 秒内不能重复发送。
- 单个邮箱每小时最多发送 5 次。
- 单个验证码最多尝试 5 次。
- 成功后立即失效。
- 数据库只保存验证码哈希。
- 对外响应避免泄露邮箱是否已注册。

限流同时考虑邮箱和来源地址，防止攻击者用单一维度绕过限制。

### 注册验证

开启注册验证后：

1. 用户提交邮箱并请求验证码。
2. 服务器校验邮箱格式和域名白名单。
3. 验证码发送成功后，用户提交注册信息和验证码。
4. 验证成功才创建账户并标记邮箱已验证。

首次安装创建管理员不受此限制，防止 SMTP 未配置时锁死系统。

现有用户升级后不会被强制退出。密码登录继续可用；现有用户首次成功使用邮箱验证码登录或完成邮箱验证后更新验证状态。

### 邮箱验证码登录

登录页保留密码登录，并增加验证码登录入口：

1. 用户输入邮箱。
2. 请求登录验证码。
3. 输入验证码。
4. 验证成功后建立正常登录会话。

OAuth 和 LDAP 不进入本地验证码流程。

### 找回密码

找回密码使用一次性链接：

- 30 分钟有效。
- 数据库只保存 Token 哈希。
- 使用一次后失效。
- 重置成功后旧登录会话失效。
- 向账户邮箱发送密码已重置安全通知。

### 敏感操作二次验证

验证成功后生成绑定当前用户和会话的 10 分钟二次验证窗口。以下操作使用该窗口：

- 修改密码。
- 修改账单地址。
- 修改登录邮箱。

修改登录邮箱需要先验证当前邮箱，再验证新邮箱。管理员代用户编辑订阅或额度不要求用户邮箱验证，但必须写管理员审计流水。

## 邮件模板与发送记录

管理员编辑：

- 邮件主题。
- Markdown 正文。
- 是否启用该业务邮件。

系统使用统一 ArtiChat 邮件外壳生成 HTML 和纯文本版本。模板只开放白名单变量，不允许执行代码或注入任意模板表达式。

0.1.3 提供以下默认模板：

- 注册验证码
- 邮箱验证码登录
- 敏感操作验证码
- 找回密码
- 密码修改成功通知
- 登录邮箱修改通知
- 账单地址修改通知
- 新订阅到账或订阅变更通知
- SMTP 测试邮件

验证码和重置邮件必须发送成功后，流程才进入等待验证状态。

订阅通知等业务邮件在业务事务提交后写入发送记录并尝试发送。发送失败不撤销已成功生效的订阅，管理员可以查看失败原因并手动重试。

## 礼品卡入口

用户侧边栏头像区域上方增加“可领取的礼品”按钮：

- 只有存在待领取礼品卡时显示。
- 不显示待领取数量。
- 点击直接打开用户设置中的“兑换码”页。
- 领取完成并刷新待领取状态后自动隐藏。
- 礼品领取继续使用现有后端原子领取逻辑。

## 数据迁移

Alembic 迁移遵循以下规则：

- 只新增表、字段和索引，不删除 0.1.2 字段。
- 保留现有用户、订阅、余额、兑换码、礼品卡、公告和历史用量。
- 新字段允许历史记录为空。
- 为旧模型策略生成等价四段价格。
- 邮箱验证状态、验证码、重置链接、发送记录和模板使用独立数据结构。
- 数据库升级失败时终止应用启动，不允许静默运行在半迁移状态。

新增结构必须兼容当前主要 SQLite 部署，并遵循现有 SQLAlchemy/Alembic 模式，避免阻断未来 PostgreSQL 部署。

## 错误处理与事务

- 批量模型策略更新使用单事务，全成或全败。
- 计费记录、余额扣减和对应余额流水使用一致事务边界。
- 礼品卡领取保持现有原子状态切换。
- 验证码发送失败时不向用户暴露有效挑战。
- 业务通知邮件失败不回滚已经提交的业务事务。
- 敏感配置、验证码和重置 Token 不写入普通日志。
- 前端为网络失败、验证过期、限流和配置错误显示可操作提示。

文件分类为：

- 运行源码和资源。
- 数据库迁移。

- 发布和部署文件。
- 许可证与必要归属文件。
