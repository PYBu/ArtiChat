## Issue 1 - Image Edit Attachment Access

### Observed behavior

- Image edit tool input can contain an internal file URL such as:
  `/api/v1/files/{id}/content`.
- A server-side edit attempt returned `400: [ERROR: Unauthorized]`.
- A model may refuse to call the edit tool when the attachment is described as a raw Base64 file because it cannot resolve that attachment to an accessible file reference.

### Confirmed behavior

- Generated images are saved by ArtiChat and exposed as protected file-content URLs.
- `/api/v1/files/{id}/content` requires a verified user and file read access; an absolute URL does not make it public.
- The built-in `edit_image` tool runs under the current user's identity, not an independent model identity.
- The tool accepts `image_urls: list[str]`; it does not automatically receive a separate attachment object.
- A complete `data:image/...;base64,...` value can be consumed directly. Raw Base64 without the data-URL prefix is treated as a file ID and cannot be resolved.
- File references are readable when the current user owns the file, is an administrator, or has an applicable read grant.
- Built-in edit-tool availability requires the image-edit setting, model image-generation capability, request feature enablement, and the user's image-generation permission.
- The OpenAI edit branch sends a single image as a multipart `image` file after local resolution. A direct upstream curl test against the configured provider returned HTTP 200.

### Suspected root cause

- The native tool-calling path does not reliably convert the current chat attachment into a tool-readable file ID or complete data URL before the model calls `edit_image`.
- The chat image URL conversion path also does not consistently normalize internal `/api/v1/files/{id}/content` references before sending them to the model.

### Planned direction

1. Preserve the existing `/api/v1/files/{id}/content` display format by default.
2. Resolve internal file references server-side before every image-edit upstream request and send image bytes as multipart data.
3. Allow complete data URLs, file IDs, and internal file-content references through one shared resolver.
4. Make the current chat's attached image available to `edit_image` without requiring the model to reproduce a large raw Base64 payload.
5. Add a 0.1.6 settings switch for absolute output URL formatting only; do not make protected file URLs public.
6. Add regression tests for generated-image edit, uploaded-image edit, raw Base64 rejection, data-URL input, access checks, and the direct edit API payload.

### Implemented

- Added `open_webui.utils.image_refs`, a shared resolver for file IDs, relative/absolute ArtiChat file URLs, complete data URLs, and external HTTPS images.
- Internal file references are resolved directly from Storage after the current user's read-access check; they never make an unauthenticated HTTP request back to ArtiChat.
- Raw Base64 without a data-URL prefix is rejected with a structured image-reference error.
- The `edit_image` tool now accepts omitted image references when the current metadata contains an image attachment, preferring its file ID.
- User and assistant history file tags now include escaped stable file IDs, so generated assistant images remain available for a later edit request without embedding raw Base64.
- Generated and edited image results now preserve the uploaded file ID alongside the protected content URL.
- OpenAI, Gemini, and ComfyUI edit paths consume the shared resolved image bytes; OpenAI receives the source as multipart data.
- Added `images.output_url_format` with `relative` as the default and an admin segmented control for `relative`/`absolute` output URLs. The setting does not change file access protection.
- Corrected the playground edit API request to send the FastAPI edit form at the request body root.

## Deferred Issue - Stable Model Provider Routing

### Observed behavior

- OpenAI-compatible Base URLs and API keys are stored in parallel arrays and paired by list index.
- Automatically discovered models are merged by their upstream model ID.
- When multiple connections expose the same model ID, the first connection wins and later models are silently discarded.
- Requests then use the cached `urlIdx` to select the endpoint and API key. Removing or reordering connections can therefore make routing difficult to understand and potentially change which credential is used.

### Agreed direction

- Give every provider connection a stable, opaque connection ID instead of using its array position as identity.
- Treat model discovery as an administrative candidate list rather than automatically publishing every discovered model.
- Bind each enabled ArtiChat model to an explicit connection ID and upstream model ID.
- Preserve same-name models from different connections as distinct choices with visible provider labels.
- Record a safe connection label, endpoint host, and non-secret key fingerprint for administrative routing audits.
- Keep optional key pooling, failover, and load balancing out of the first routing fix; the default must remain one deterministic connection and credential.

### Scope decision

- The issue is recorded for a later implementation phase.
- Stable model-provider routing remains outside 0.1.6; this release includes the accepted image-edit and gift-card/admin improvements above.
